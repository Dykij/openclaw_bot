# ROADMAP_OPENCLAW2026: Глобальная архитектура и ИИ-оркестрация

Этот документ определяет вектор развития системы **OpenClaw** на 2026 год, основываясь на принципах агентно-ориентированной разработки, максимальной безопасности и эффективного использования локального ИИ.

## Фаза 1: Инфраструктура и Безопасность (Q1-Q2 2026)

### 1.1 Rootless Docker & Hardening
- [x] **Миграция на Rootless Mode**: Полный перевод всех контейнеров OpenClaw на запуск от имени непривилегированного пользователя для исключения рисков Container Escape.
- [x] **Директива Least Privilege**: Внедрение в `docker-compose.yml` строгих ограничений (`--cap-drop ALL`, `--read-only`, `no-new-privileges: true`).
- [x] **Alpine/Distroless Migration**: Замена базовых образов на сверхлегкие дистрибутивы (multi-stage runtime) для минимизации поверхности атаки.

### 1.2 Изоляция ИИ-кода через gVisor
- [x] **Runtime Isolation**: Настройка `runsc` (gVisor) для всех воркеров, исполняющих динамически генерируемый ИИ-код (Гейтвей и Сайнер).
- [x] **Seccomp & Filter**: Разработка кастомных профилей Seccomp для блокировки всех системных вызовов, не связанных с сетевым взаимодействием и базовыми операциями памяти.

---

## Фаза 2: Оркестрация и STAR-логика (Q2-Q3 2026)

### 2.1 Глубокая интеграция STAR Framework
- [x] **Native Reasoning**: Внедрение структуры **Situation-Task-Action-Result** непосредственно в ядро `PipelineExecutor`. 
- [x] **Reasoning Scaffolding**: Принудительное использование тегов `<think>` для моделей класса DeepSeek-R1 и Gemini 3.1 Pro перед любым вызовом инструментов (tools).

### 2.2 Многоуровневая память (Memory Bank)
- [x] **RAG Hierarchy**: Оптимизация разделения на `Hot_Memory` (текущая сессия), `Domain_Experts` (база знаний по API/Трейдингу) и `Cold_Memory` (исторические решения).
- [x] **BM25 + Vector Search**: Переход на гибридный поиск для повышения точности извлечения фактов и подавления галлюцинаций.

### 2.3 MCP Evolution
- [x] **Unified Interface**: Перевод всех расширений OpenClaw на стандарт **Model Context Protocol (MCP)**.
- [x] **Self-Hosted Gateway**: Оптимизация работы воркеров за NAT/Firewall через поллинг задач, исключая внешние порты.

---

## Фаза 3: Аппаратная оптимизация (Q3-Q4 2026)

### 3.1 VRAM & Multi-Model Loading
- [x] **Smart Swapping Logic**: Автоматизация выгрузки (`keep_alive=0`) и очистки CUDA-кэша при переключении между тяжелыми моделями (arkady-27b <-> deepseek-r1-14b).
- [x] **Quantization Strategy**: Стандартизация использования 4-битных квантованных моделей (GGUF/NVFP4) для полной утилизации RTX 5060 Ti 16GB.

### 3.2 Performance Monitoring
- [x] **Latency Dashboard**: Визуализация задержек принятия решений ИИ-агентами по сравнению с рыночными событиями.
- [x] **Token Efficiency Auditor**: Автоматический анализ выхлопа моделей на предмет лишнего текста (отсекание "болтовни" ИИ).

---

## Фаза 4: Надежность и WSL (Q4 2026 - Завершено досрочно)

### 4.1 WSL Native Execution
- [x] **WSL Integration**: Полный переход на запуск Гейтвея внутри WSL для стабильности и совместимости с Linux-библиотеками.
- [x] **Cross-Env Connectivity**: Настройка сетевого взаимодействия между WSL и Windows-хостом для Ollama и Task Server.

### 4.2 Reliability & Bug Fixes
- [x] **Telegram Conflict Resolution**: Внедрение агрессивного управления PID и блокировками для предотвращения запуска нескольких инстансов бота.
- [x] **Polling Deduplication**: Реализация хеширования задач (SHA-256) в `main.py` для защиты от бесконечных циклов в Polling Gateway.

## Методология разработки
- **Spec-driven development**: Использование Markdown-спецификаций как основного источника истины для генерации кода.
- **Agent-first workflow**: Использование Google Antigravity и `Mission Control` для параллельного рефакторинга и тестирования.
