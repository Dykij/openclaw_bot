# Анализ Gemini: оптимизация инференса LLM

## Оценка правдоподобности утверждений

### 1. Memory-bound природа авторегрессионного инференса ✅ ПОДТВЕРЖДЕНО

**Утверждение:** При декодировании каждого токена GPU обязан прочитать все веса модели из HBM в SRAM. Для модели 70B в FP16 это ~140 ГБ на токен. Декодирование жёстко ограничено пропускной способностью памяти.

**Вердикт:** Полностью корректно. Это фундаментальное свойство трансформерного декодирования. Для RTX 5060 Ti (Blackwell GB206, 16 GB GDDR7) пропускная способность памяти составляет ~672 ГБ/с. Модель Qwen-14B-AWQ (~7 ГБ в AWQ) читается за ~10 мс на токен, что даёт теоретический потолок ~100 TPS для одного запроса (без батчинга). Реальные цифры ниже из-за накладных расходов.

---

### 2. Непрерывное пакетирование (Continuous Batching), 2–10× TPS ✅ ПОДТВЕРЖДЕНО (с оговоркой)

**Утверждение:** Continuous Batching увеличивает пропускную способность в 2–10 раз. Алгоритм был формализован в системе Orca.

**Вердикт:** Диапазон 2–10× реалистичен для *пропускной способности* (Throughput) при высокой нагрузке, но не для латентности одного запроса. Orca — правильная ссылка (Yu et al., 2022). vLLM использует continuous batching по умолчанию. В этом репозитории vLLM уже запускается с непрерывным батчингом — никаких дополнительных флагов не требуется.

---

### 3. PagedAttention и фрагментация памяти ✅ ПОДТВЕРЖДЕНО

**Утверждение:** Традиционный KV-кэш выделяет непрерывный блок на максимальную длину последовательности, вызывая внутреннюю и внешнюю фрагментацию. PagedAttention решает это через страничную виртуальную память.

**Вердикт:** Точно описывает вклад статьи vLLM (Kwon et al., 2023, arXiv:2309.06180). vLLM использует PagedAttention по умолчанию, это прозрачно для пользователя. Реализовано в репозитории через vLLM.

---

### 4. RadixAttention и кэширование префиксов, 3–5× снижение TTFT ✅ ПОДТВЕРЖДЕНО

**Утверждение:** RadixAttention (SGLang) сохраняет KV-кэш в префиксном дереве на CPU, позволяя повторно использовать его между запросами. Накладные расходы на CPU — ~0.07 с для 128 запросов. TTFT снижается в 3–5 раз.

**Вердикт:** Корректно. SGLang RadixAttention (arXiv:2312.07104) — реальная система. vLLM реализовал эквивалентную функциональность через `--enable-prefix-caching`. Для этого бота (системный промпт + контекст Deep Research повторяется в каждом запросе) это наиболее высокоприоритетная оптимизация.

**Что реализовано:** Класс `PrefixCachingConfig` в `src/inference_optimizer.py` — активируется флагом `--enable-prefix-caching`.

---

### 5. Chunked Prefill (контроль ITL) ✅ ПОДТВЕРЖДЕНО

**Утверждение:** Длинные промпты (32k токенов) монополизируют GPU на сотни мс. Chunked Prefill разбивает промпт на чанки, чередуя их с декодированием.

**Вердикт:** Точно описывает Sarathi-Serve (Agrawal et al., 2023, arXiv:2308.16369). Параметр `max_num_batched_tokens=2048` для гладкого ITL vs `8192` для низкого TTFT — правильный trade-off.

**Что реализовано:** Класс `ChunkedPrefillConfig` в `src/inference_optimizer.py`.

---

### 6. Спекулятивное декодирование, 1.5–2.5× ускорение ✅ ПОДТВЕРЖДЕНО (с важным уточнением)

**Утверждение:** Draft-модель генерирует K токенов, целевая модель верифицирует их за один проход. Ускорение 1.5–2.5×. Гарантия lossless — итоговое распределение идентично исходному.

**Вердикт:** Математическая гарантия lossless (через rejection sampling) — корректна (Leviathan et al., 2022, arXiv:2211.17192). Диапазон 1.5–2.5× реалистичен для текстов с хорошей предсказуемостью (код, диалог, документация).

**Ключевое упущение в анализе Gemini:** Для системы с 16 GB VRAM использование Draft-модели неоптимально, поскольку требует ~1 ГБ дополнительной VRAM. **N-gram спекулятивное декодирование** (`--speculative-model [ngram]`) решает эту проблему полностью: черновики генерируются путём поиска подстрок в самом промпте, без отдельной модели, и без затрат VRAM. Для типичных задач бота (RAG, повторяющийся контекст) ускорение составляет 1.3–2×.

**Что реализовано:** `SpeculativeDecodingConfig` расширен: `use_ngram=True` (по умолчанию) активирует режим без дополнительной VRAM.

---

### 7. Утверждение о 100% верификации vs последовательной генерации ✅ ПОДТВЕРЖДЕНО

**Утверждение:** GPU за одно и то же wall-clock время обрабатывает и один токен, и маленький батч из K токенов.

**Вердикт:** Корректно при K ≤ порога насыщения шины (обычно 10–20 токенов на GPU уровня RTX серии). Именно это делает спекулятивное декодирование возможным без регрессии.

## Реализованные улучшения в репозитории

### `src/inference_optimizer.py`

| Класс / функция | Описание | Ожидаемый эффект |
|---|---|---|
| `SpeculativeDecodingConfig` (обновлён) | N-gram режим (`use_ngram=True`) + Draft-model режим | 1.3–2.5× TPS, 0 дополнительной VRAM (ngram) |
| `ChunkedPrefillConfig` (новый) | `--enable-chunked-prefill --max-num-batched-tokens N` | Гладкий ITL при длинных промптах Deep Research |
| `PrefixCachingConfig` (новый) | `--enable-prefix-caching` | TTFT ÷ 3–5 для повторяющихся системных промптов |
| `build_optimized_vllm_args()` (новая) | Компонует флаги всех трёх конфигов | Единая точка сборки аргументов vLLM |

### `src/vllm_manager.py`

`VLLMModelManager.__init__` принимает `speculative`, `chunked_prefill`, `prefix_caching` — опциональные параметры конфигурации. Флаги автоматически передаются в vLLM при запуске сервера.

## Рекомендуемые конфигурации для RTX 5060 Ti (16 GB)

### Конфигурация А: максимальный TPS при низкой задержке (рекомендуется)

```python
from src.inference_optimizer import (
    SpeculativeDecodingConfig,
    PrefixCachingConfig,
    ChunkedPrefillConfig,
)
from src.vllm_manager import VLLMModelManager

manager = VLLMModelManager(
    port=8000,
    gpu_memory_utilization=0.92,
    max_model_len=16384,
    quantization="awq",
    # N-gram спекулятивное декодирование — нулевые затраты VRAM
    speculative=SpeculativeDecodingConfig(
        enabled=True,
        use_ngram=True,
        ngram_prompt_lookup_max=4,
        num_speculative_tokens=5,
    ),
    # Кэширование префиксов — критично для системных промптов бота
    prefix_caching=PrefixCachingConfig(enabled=True),
    # Chunked prefill — для гладкого ITL при Deep Research контексте
    chunked_prefill=ChunkedPrefillConfig(
        enabled=True,
        max_num_batched_tokens=4096,
    ),
)
```

**Ожидаемый совокупный эффект:**
- TPS: +30–100% (ngram spec decoding + prefix cache hits)
- TTFT: ÷ 3–5× (prefix caching для системного промпта)
- ITL сглаживается (chunked prefill)
- Дополнительная VRAM: 0 ГБ

### Конфигурация Б: максимальное качество (сложные задачи)

```python
manager = VLLMModelManager(
    speculative=SpeculativeDecodingConfig(enabled=False),
    prefix_caching=PrefixCachingConfig(enabled=True),
    chunked_prefill=ChunkedPrefillConfig(
        enabled=True,
        max_num_batched_tokens=8192,
    ),
)
```

## Дополнительные улучшения, не рассмотренные в анализе Gemini

### 1. Оптимальное использование AWQ-квантования (уже реализовано)

AWQ (Activation-aware Weight Quantization, arXiv:2306.00978) сжимает веса до INT4 с минимальной потерей качества. Модели Qwen-14B-AWQ и DeepSeek-R1-14B-AWQ уже в репозитории — это правильный выбор для 16 GB VRAM.

### 2. `--max-num-seqs` — ограничение concurrent sequences

Флаг `--max-num-seqs 256` (или выше) увеличивает количество одновременно обрабатываемых запросов. По умолчанию vLLM использует 256, но при AWQ-моделях это значение можно увеличить до 512–1024, так как меньший footprint VRAM оставляет больше памяти для KV-кэша.

### 3. Flash Attention 2 / Flash Attention 3

Blackwell GB206 поддерживает Flash Attention 2 автоматически через vLLM. Flash Attention 3 (для Hopper H100) нерелевантен для RTX 5060 Ti, но FA2 даёт ~20% ускорение на фазе Prefill.

### 4. CUDA Graph (уже частично реализовано в `llm_engine.py`)

`enforce_eager=True` в `llm_engine.py` отключает CUDA-граф. Для production следует убрать этот флаг (или установить `enforce_eager=False`) — это даст 10–20% ускорение за счёт компиляции статического графа выполнения.

## Итоговая оценка анализа Gemini

Анализ **технически точен** в описании основных концепций (Memory-bound декодирование, Continuous Batching, PagedAttention, Chunked Prefill, Speculative Decoding). Цифры производительности реалистичны и подкреплены публикациями.

**Главное упущение:** Анализ не упоминает n-gram спекулятивное декодирование — ключевую альтернативу для систем с ограниченной VRAM. Использование Draft-модели (как предлагается в тексте) неоптимально для однокарточных систем с 16 ГБ, таких как RTX 5060 Ti.

**Реализованные улучшения** покрывают три наиболее высокодоходные техники для данного железа:
1. N-gram спекулятивное декодирование (нулевые затраты VRAM, +30–100% TPS)
2. Prefix caching (÷ 3–5 TTFT для повторяющихся промптов)
3. Chunked prefill (гладкий ITL при Deep Research контексте)
