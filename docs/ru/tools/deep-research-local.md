# Deep Research на локальной модели

Этот гайд описывает как выполнять глубокий анализ (Deep Research) документации и информации используя **только локальную модель** через Ollama — без зависимости от внешних API, сторонних сервисов и подключения к Интернету.

## Что такое локальный Deep Research

Локальный Deep Research — это многоэтапный анализ информации, где агент OpenClaw:

1. **Декомпозирует** вопрос на подзадачи
2. **Ищет** по индексированной документации через `memory_search`
3. **Читает** найденные документы через `memory_get`
4. **Перекрёстно проверяет** данные из разных источников
5. **Синтезирует** полный ответ со ссылками на источники

Всё это работает офлайн, на одной GPU (даже на AMD RX 6600 с 8GB VRAM).

## Архитектура локального Deep Research

```
Вопрос пользователя
  → Декомпозиция на подвопросы (LLM)
  → memory_search по каждому подвопросу
  → memory_get для чтения найденных файлов
  → Перекрёстная проверка (LLM)
  → Синтез ответа со ссылками
  → Финальный ответ
```

### Компоненты системы

| Компонент | Роль | Работает офлайн |
| --------- | ---- | --------------- |
| **Ollama** | Инференс LLM (qwen2.5-coder, deepseek-r1, llama3.3) | ✅ Полностью |
| **memory_search** | Семантический поиск по документам workspace | ✅ С локальными эмбеддингами |
| **memory_get** | Чтение файлов из memory/ и MEMORY.md | ✅ Полностью |
| **openclaw docs** | Поиск по индексу docs.openclaw.ai | ❌ Нужен интернет |
| **web_search** | Поиск в интернете | ❌ Нужен интернет |
| **doc_ingester.py** | Индексация внешних страниц в .memory-bank/ | ⚠️ Нужен интернет для загрузки |

## Настройка локального Deep Research

### Шаг 1: Установить Ollama и модель

```bash
# Установить Ollama
curl -fsSL https://ollama.ai/install.sh | sh

# Загрузить модель (выберите под вашу VRAM)
ollama pull qwen2.5-coder:7b    # 4.4GB — хорошо для 8GB VRAM
ollama pull deepseek-r1:8b      # 4.9GB — reasoning модель
ollama pull llama3.3:8b          # 4.9GB — общего назначения
```

### Шаг 2: Настроить OpenClaw для локальной работы

```json5
{
  agents: {
    defaults: {
      // Основная модель через Ollama
      model: { primary: "ollama/qwen2.5-coder:7b" },

      // Память — ключевой компонент Deep Research
      memorySearch: {
        enabled: true,
        // Локальные эмбеддинги (без внешних API)
        provider: "local",
        sources: ["memory", "sessions"],
        chunking: {
          tokens: 400,    // Размер чанка
          overlap: 80,    // Перекрытие для контекста
        },
        query: {
          maxResults: 10,
          minScore: 0.3,
          hybrid: {
            enabled: true,
            vectorWeight: 0.6,
            textWeight: 0.4,
          },
        },
      },
    },
  },
  models: {
    providers: {
      ollama: {
        apiKey: "ollama-local",
        baseUrl: "http://127.0.0.1:11434",
      },
    },
  },
}
```

### Шаг 3: Подготовить базу знаний

Создайте документы в workspace агента:

```bash
# Структура workspace
~/.openclaw/agents/default/
├── MEMORY.md          # Долгосрочная память (курируемые факты)
├── memory/
│   ├── 2026-03-17.md  # Ежедневный лог
│   ├── architecture.md # Архитектурные решения
│   └── research/      # Результаты исследований
└── AGENTS.md          # Конфигурация агента
```

#### Формат MEMORY.md

```markdown
# Долгосрочная память

## Архитектура
- OpenClaw Gateway работает как единый daemon на порту 18789
- Поддерживаемые каналы: WhatsApp, Telegram, Discord, Slack, Signal, iMessage
- Маршрутизация сообщений через channel adapters

## Конфигурация
- Конфиг: ~/.openclaw/openclaw.json
- Для Ollama: models.providers.ollama.apiKey = "ollama-local"
- VRAM-менеджмент: keep_alive=0 для выгрузки после использования

## Принятые решения
- [2026-03-05] Один физический LLM в VRAM, переключение через System Prompts
- [2026-03-17] Внедрение локального Deep Research через memory_search
```

### Шаг 4: Индексировать документацию офлайн

Для индексации документации OpenClaw без интернета, скопируйте docs/ в workspace:

```bash
# Скопировать документацию в memory-банк агента
cp -r docs/ ~/.openclaw/agents/default/memory/docs/

# Или использовать doc_ingester для внешних источников (один раз, при наличии интернета)
python scripts/doc_ingester.py --url https://docs.openclaw.ai/concepts/architecture
python scripts/doc_ingester.py --url https://docs.openclaw.ai/concepts/memory
```

После индексации все данные хранятся локально в `.memory-bank/docs/` и доступны офлайн.

## Оптимизация для ограниченной VRAM

### Управление VRAM на AMD RX 6600 (8GB)

```json5
{
  agents: {
    defaults: {
      model: {
        primary: "ollama/qwen2.5-coder:7b",
        // НЕ загружать модель в VRAM постоянно
        // OpenClaw отправляет keep_alive=0 после каждого цикла
      },
      // Уменьшить контекстное окно для экономии VRAM
      memorySearch: {
        chunking: {
          tokens: 300,   // Меньшие чанки = меньше VRAM
          overlap: 60,
        },
        query: {
          maxResults: 5,  // Меньше результатов = меньше контекста
        },
      },
    },
  },
}
```

### Выбор модели по задаче

| Задача | Рекомендуемая модель | VRAM | Контекст |
| ------ | -------------------- | ---- | -------- |
| Анализ кода | `qwen2.5-coder:7b` | ~4.4GB | 32K |
| Рассуждения и логика | `deepseek-r1:8b` | ~4.9GB | 64K |
| Общие вопросы | `llama3.3:8b` | ~4.9GB | 128K |
| Быстрые ответы | `qwen2.5:1.5b` | ~0.9GB | 32K |
| Классификация | `qwen2.5:0.5b` | ~0.3GB | 32K |

### Стратегия переключения моделей

При 8GB VRAM в памяти может находиться только одна модель. Используйте стратегию из Hot Memory:

```
1. Загрузить основную модель (qwen2.5-coder:7b) для анализа
2. Выполнить Deep Research (memory_search → memory_get → синтез)
3. Выгрузить модель (keep_alive=0)
4. При необходимости загрузить reasoning модель (deepseek-r1:8b)
5. Выполнить перекрёстную проверку результатов
6. Выгрузить модель
```

Для оптимального поиска через `memory_search`, структурируйте документы так:

````markdown
---
summary: "Краткое описание содержания (одно предложение)"

  - Ещё один сценарий

# Чёткий заголовок, описывающий тему

Первый абзац — самая важная информация.
AI модели часто обрезают контекст, поэтому ключевые факты должны быть в начале.

## Подраздел с описательным заголовком

Каждый раздел H2 должен быть понятен отдельно от остальной страницы,
так как RAG-системы могут извлечь именно эту секцию.

### Код с тегами языка

```bash
# Всегда указывайте язык для подсветки
openclaw config set models.providers.ollama.apiKey "ollama-local"
```

## FAQ

### Часто задаваемый вопрос?

Прямой, конкретный ответ. FAQ-формат хорошо подходит для поиска.
````

### Принципы читаемой документации

1. **Ответ вначале** — Начинайте каждый раздел с ключевого факта или ответа
2. **Один топик — одна страница** — Не смешивайте разные темы на одной странице
3. **Описательные заголовки** — Вместо "Настройка" пишите "Как настроить Telegram канал"
4. **Самодостаточные секции** — Каждый H2 раздел должен быть понятен без контекста
5. **Таблицы для сравнений** — Таблицы парсятся надёжнее, чем текстовые описания
6. **Примеры кода с тегами** — Всегда указывайте язык: ` ```bash `, ` ```json5 `, ` ```typescript `
7. **Избегайте местоимений** — Пишите "Gateway" вместо "он" в технической документации
8. **Перекрёстные ссылки** — Всегда ссылайтесь на связанные документы

## Локальный RAG-пайплайн

### Как работает memory_search

1. Документы из `memory/` и `MEMORY.md` разбиваются на чанки (~400 токенов, 80 токенов перекрытие)
2. Каждый чанк индексируется через эмбеддинг-модель (локальную или удалённую)
3. При запросе: вычисляется эмбеддинг запроса → cosine similarity → top-K результатов
4. Гибридный поиск комбинирует векторный (семантический) + текстовый (FTS5) поиск

### Схема индексации

```
Файлы workspace:
  MEMORY.md
  memory/*.md
      ↓
  Chunking (400 токенов, 80 перекрытие)
      ↓
  SQLite + sqlite-vec (векторная БД)
      ↓
  memory_search → Top-K результаты
      ↓
  memory_get → Полное содержимое файла
```

### Настройка локальных эмбеддингов

Для полностью офлайн работы без внешних API используйте локальные эмбеддинги:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        enabled: true,
        provider: "local",
        local: {
          // Путь к модели эмбеддингов (GGUF формат)
          // modelPath: "/path/to/embedding-model.gguf",
        },
        store: {
          driver: "sqlite",
          vector: {
            enabled: true,
            // extensionPath: "/path/to/vec0.so",  // При необходимости
          },
        },
      },
    },
  },
}
```

## Устранение проблем

### memory_search не находит результаты

- Проверьте что файлы существуют в `memory/` или `MEMORY.md`
- Запустите `openclaw doctor` — проверит доступность эмбеддингов
- Убедитесь что `memorySearch.enabled: true` в конфиге
- Попробуйте увеличить `query.maxResults` и уменьшить `query.minScore`

### Медленная индексация

- Локальные эмбеддинги работают на CPU — это нормально для первой индексации
- Последующие обновления инкрементальны и быстрее
- Уменьшите `chunking.tokens` для быстрого индексирования (но с потерей контекста)

### Ollama не отвечает

```bash
# Проверить статус
systemctl status ollama
# Или
ollama list

# Перезапустить при необходимости
systemctl restart ollama
```

### Нехватка VRAM

- Используйте модели меньшего размера: `qwen2.5:1.5b` (0.9GB VRAM)
- Установите `keep_alive: 0` для немедленной выгрузки модели
- Закройте другие GPU-приложения
